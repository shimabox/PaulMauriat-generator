# PaulMauriat-generator ソースコード解析

解析日: 2026-07-13

対象: `index.html`、`css/style.css`、`js/generator.js`、`js/v2c.js`、`README.md`、同梱ライブラリ

## 1. エグゼクティブサマリー

このリポジトリは、選択した背景画像にカメラから切り出した顔を重ね、PNG として保存するブラウザ完結型の静的 Web アプリである。サーバー処理もビルド工程も不要で、機能の意図が分かりやすく、EXIF の向き補正、前面カメラの反転、顔の配置・透明度・プライバシー加工まで一通り実装されている点はよい。

一方、最優先で見直したいのは次の3点である。

1. **画像読み込みの堅牢性**: JPEG 前提の EXIF 解析をすべての `image/*` に適用しており、短い PNG/GIF などで `RangeError` が発生する。
2. **カメラのライフサイクル**: Stop 操作が描画ループだけを止め、MediaStream のトラックを停止しないため、カメラが使用中のまま残る。
3. **変更容易性と回帰防止**: 主要処理が682行の1ファイルに集中し、自動テスト、依存バージョン管理、CI がない。

総評として、**小規模な作品としては目的に対して素直で完成度が高いが、現在のブラウザ環境で継続保守するには、入力境界・状態管理・テスト基盤を補強したいコード**である。

## 2. 構成と処理フロー

| ファイル | 主な責務 |
| --- | --- |
| `index.html` | 画像選択、開始・停止・保存・カメラ切替、顔表示設定の UI |
| `css/style.css` | 画面レイアウト、表示・非表示、操作状態の見た目 |
| `js/generator.js` | UI イベント、背景Canvasへの向き補正、顔領域の切り出し・加工・合成、PNG 出力 |
| `js/image-loader.js` | 画像ファイルの検証・読み込み、寸法上限確認、EXIF取得、画像デコード、Object URL解放 |
| `js/v2c.js` | `getUserMedia`、video/canvas の生成、カメラ切替、描画ループ |
| `js/vendor/clmtrackr/clmtrackr.min.js` | 顔特徴点の検出・追跡 |

処理は概ね次の順に進む。

1. ユーザーが画像を選択またはドロップする。
2. `ImageLoader` で画像を検証・解析・デコードし、EXIF Orientation に応じて背景用 Canvas に描く。
3. `getUserMedia` でカメラを開始し、映像を別の Canvas に毎フレーム転写する。
4. `clmtrackr` の特徴点から顔・目の矩形を計算する。
5. 目線またはモザイクを任意で適用し、円形・半透明の顔 Canvas を背景上に配置する。
6. 背景と顔を出力用 Canvas に再合成し、PNG の Data URL として保存する。

## 3. いいところ

### 3.1 ブラウザ内で処理が完結している

選択画像とカメラフレームを外部へ送信する処理は見当たらず、Canvas 上で加工・保存している。個人の顔や手元の画像を扱うアプリとして、データフローが小さく理解しやすい。静的ホスティングだけで公開できるため、運用コストと攻撃面も抑えられている。

### 3.2 機能がユーザー操作に沿って一通り揃っている

画像の再選択、顔追跡の開始・停止、PNG 保存、前面・背面カメラ切替、4隅への配置、透明度変更、目線・モザイク加工が、一つの画面で完結している（[`index.html`](index.html#L27-L63)）。README に画面番号と操作方法も記載されており、初見の利用者が機能を把握しやすい（[`README.md`](README.md#L12-L40)）。

### 3.3 EXIF Orientation の8パターンを描画変換に反映している

Orientation 2〜8に応じた反転・回転を `CanvasRenderingContext2D.transform()` で処理しており、スマートフォン写真を意識した実装になっている（[`js/generator.js`](js/generator.js#L175-L201)）。EXIF の読み取り部分には改善余地があるものの、画像の向きを補正する設計自体は適切である。

### 3.4 カメラ処理を `V2C` に分離している

カメラの取得、video/canvas の生成、描画ループ、カメラ切替が `V2C` に集められている（[`js/v2c.js`](js/v2c.js#L36-L108)）。アプリ固有の顔合成処理とブラウザ API の詳細を分けようとしており、今後コントローラーとして整理する土台がある。

### 3.5 前面カメラのミラー表示と保存結果を考慮している

プレビューでは CSS の `scaleX(-1)` を使い、保存時は出力 Canvas に描き直して反転を実体化している（[`js/generator.js`](js/generator.js#L330-L340)、[`js/generator.js`](js/generator.js#L635-L650)）。見た目だけ反転して保存結果が逆になる、という Canvas アプリで起こりやすい不整合を避けている。

### 3.6 顔が大きすぎる場合の縮小や画面端を考慮している

顔領域にマージンを加えつつ、背景画像の短辺の3分の1を上限に縮小している（[`js/generator.js`](js/generator.js#L361-L433)）。また、特徴点が映像外に出たときは描画を更新しないガードもあり、顔合成の見た目を安定させようとする工夫がある。

## 4. 懸念点

優先度は、ユーザー影響、再現しやすさ、プライバシー・保守性への影響を合わせて判断した。

### 4.1 高: JPEG 以外や壊れた画像で EXIF 解析が例外になりうる

**根拠**

- 入力は MIME が `image.*` に一致すれば許可される（[`js/generator.js`](js/generator.js#L110-L117)）。
- その直後、形式を判定せず全ファイルを `getOrientation()` に渡す（[`js/generator.js`](js/generator.js#L119-L135)）。
- `getOrientation()` は JPEG の SOI、各セグメント長、バッファ終端を検証せず、固定位置またはファイル内容由来の位置を `DataView` で読む（[`js/generator.js`](js/generator.js#L139-L166)）。

確認用に PNG/GIF のシグネチャを同じロジックへ渡すと、どちらも `RangeError: Offset is outside the bounds of the DataView` になった。実ファイルでも、短い非 JPEG 画像や不正・途中で切れた JPEG で同様の停止が起きうる。README は単に「image」と案内しているため、利用者の期待ともずれる。

**改善案**

- JPEG の SOI (`0xFFD8`) を最初に確認し、JPEG 以外は Orientation 1 として扱う。
- APP1 を固定位置で想定せず、JPEG セグメントを長さ検証しながら走査する。
- すべての読み取り前に `byteLength` とオフセットを検証し、失敗時は回転なしへフォールバックする。
- `FileReader#error`、画像の `error` イベントも処理し、画面内に利用者向けエラーを表示する。

### 4.2 高: Stop してもカメラトラックが解放されない

**根拠**

`V2C.stop()` は `requestAnimationFrame` のキャンセルと状態フラグの変更だけで、`videoTrack.stop()` や `video.srcObject = null` を行わない（[`js/v2c.js`](js/v2c.js#L71-L75)）。トラックを明示的に停止しているのはカメラ切替時だけである（[`js/v2c.js`](js/v2c.js#L76-L83)）。

このため Stop 後も端末のカメラ使用表示が残り、電力消費とプライバシー上の不安につながる。また、Stop 後の Start は新しい `getUserMedia()` を呼ぶため、古いトラックを保持したまま新しいストリームへ差し替える可能性がある。

**改善案**

- 停止時に `video.srcObject` の全トラックへ `track.stop()` を呼び、`srcObject` と `videoTrack` を `null` にする。
- 「顔の表示だけを一時停止」と「カメラ自体を終了」を別操作にしたい場合は、メソッド名と UI を `pause` / `stopCamera` のように明確に分ける。
- `pagehide` / `visibilitychange` も考慮し、ページ離脱時に確実に解放する。

### 4.3 高: 自動テストと継続的チェックがない

テストファイル、依存マニフェスト、CI 設定がなく、現状確認できる自動チェックは JavaScript の構文確認程度である。特に EXIF、座標計算、カメラ状態遷移は境界条件が多く、手動確認だけでは変更時の回帰を見つけにくい。

**改善案**

- まず副作用のない関数（EXIF 解析、矩形のクランプ、拡大率、配置座標）を別モジュールへ切り出し、表形式の単体テストを追加する。
- カメラ API を薄いアダプター越しに利用し、許可・拒否・未対応・二重 Start・Stop・切替を疑似 MediaStream でテストする。
- 実ブラウザのテストで画像選択から PNG 出力までの主要導線を確認する。
- CI で構文、静的解析、単体テストを実行する。

### 4.4 中: カメラ開始処理に状態競合と未対応環境への弱さがある

`start()` の二重実行防止は `drawLoopFrame` だけを見ているが、この値が設定されるのは `getUserMedia()` 成功後である（[`js/v2c.js`](js/v2c.js#L59-L69)）。許可ダイアログの表示中に複数回 Start すると、複数の取得要求が並行する余地がある。

また、`navigator.mediaDevices` がない環境では `_loadVideo()` が同期的に例外を投げ、後続の `.catch()` では捕捉できない（[`js/v2c.js`](js/v2c.js#L215-L220)）。背面カメラに `facingMode: { exact: 'environment' }` を要求するため、該当カメラがない端末では切替が必ず失敗する（[`js/v2c.js`](js/v2c.js#L9-L11)）。

**改善案**

- `idle` / `starting` / `running` / `paused` / `error` の明示的な状態を持ち、`starting` 中は操作を無効化する。
- `_loadVideo()` を常に Promise として失敗させられる形にし、API の存在と Secure Context を事前確認する。
- 背面カメラは `exact` 失敗時に `ideal: 'environment'`、さらに既定カメラへ段階的にフォールバックする。
- エラーは `NotAllowedError` などの種類に応じて、復旧方法を画面内に表示する。

### 4.5 中: モザイクの端数ブロック処理が領域外まで回り込む

モザイクは16px単位で処理するが、幅・高さが16の倍数でない最後のブロックでも常に16×16回書き込む（[`js/generator.js`](js/generator.js#L539-L563)）。右端で `x + x2 >= cw` になると、一次元の添字計算上は次の行へ回り込み、本来のブロック外の画素まで上書きする。下端では TypedArray の範囲外書き込みが無視されるため、端の見た目も不均一になる。

また、目の矩形には左右10px・上下5pxを無条件で追加している（[`js/generator.js`](js/generator.js#L512-L519)）。Canvas 外の読み取り自体は透明な黒として扱われるが、加工対象を明示的に Canvas 内へ正規化した方が、端での結果を予測しやすい。

**改善案**

加工矩形を `x >= 0`、`y >= 0`、`x + width <= canvas.width`、`y + height <= canvas.height` にクランプし、幅または高さが0なら加工をスキップする。内側ループは `Math.min(size, cw - x)` と `Math.min(size, ch - y)` までに制限する。これらは純粋関数にして境界値テストを追加する。

### 4.6 中: 主要処理が `generator.js` に集中している

`generator.js` は682行あり、DOM 取得、イベント登録、画像デコード、EXIF、カメラ連携、特徴点計算、プライバシー加工、配置、ダウンロードが一つの `DOMContentLoaded` クロージャに入っている。共有する可変状態も多く、個々の処理を単独でテストしにくい。

**改善案**

次の単位を目安に ES Modules へ分割する。

- `image-loader`: 形式検証、EXIF、デコード、縮小
- `camera-controller`: MediaStream と状態遷移
- `face-geometry`: 特徴点から顔・目の矩形を計算
- `face-renderer`: 切り抜き、透明度、目線・モザイク
- `exporter`: 背景と顔の合成、PNG 出力
- `app`: DOM と各モジュールの接続

### 4.7 中: 依存ライブラリの由来と更新方法が分からない

`clmtrackr.min.js` は約1.9MBの生成済みファイルとして直接コミットされているが、バージョン、入手元、ビルド方法、ライセンス情報を示すマニフェストや文書がない。機能はオフラインでも動かしやすい一方、脆弱性確認、更新、再現可能な取得が難しい。

**改善案**

- バージョン、配布元 URL、ライセンス、ファイルのハッシュ、更新手順を文書化する。
- 依存管理を導入する場合はバージョンを固定し、配布物をどう生成するかを README に記載する。
- 顔追跡実装を将来差し替えられるよう、特徴点取得をインターフェースで隔離する。

### 4.8 中: 大きな画像でメモリ使用量が増えやすい

画像全体を `FileReader` の ArrayBuffer として読み、さらに Blob、Image、Canvas を作る（[`js/generator.js`](js/generator.js#L119-L135)、[`js/generator.js`](js/generator.js#L169-L201)）。表示時の Canvas は縮小されるが、元画像は先にフルサイズでデコードされる。高解像度写真では一時的に大きなメモリを使用し、モバイルでタブが落ちる可能性がある。

**改善案**

- ファイルサイズと画像寸法に上限を設け、超過時は説明を表示する。
- 可能ならデコード時のリサイズを利用し、中間オブジェクトの寿命を短くする。
- Object URL は `load` だけでなく `error` 時にも revoke する。

### 4.9 中: レスポンシブ対応が初期サイズに依存している

背景画像の縮小基準は初期表示時に一度保存した `.view` の寸法であり（[`js/generator.js`](js/generator.js#L32-L40)、[`js/generator.js`](js/generator.js#L243-L253)）、通常のリサイズでは再計算されない。`orientationchange` ではカメラ Canvas の比率だけを調整しており、背景と顔の配置をまとめて再レイアウトしていない（[`js/v2c.js`](js/v2c.js#L193-L213)）。端末回転やウィンドウ幅変更後に、重なりや余白が不自然になる可能性がある。

**改善案**

`ResizeObserver` 等でコンテナ変更を検知し、背景・カメラ・顔の表示倍率と配置を同じ座標系から再計算する。画像の論理座標と CSS 表示サイズを分離すると、保存解像度を維持しながら表示だけレスポンシブにできる。

### 4.10 中: アクセシビリティと HTML の基本情報に不足がある

- `<html lang>` がない（[`index.html`](index.html#L1-L3)）。
- description の引用符が ASCII の `"` ではなくスマートクォートで、正しい属性として解釈されない（[`index.html`](index.html#L7)）。
- ファイル input に `accept="image/*"` がない（[`index.html`](index.html#L29)）。
- 画像ドロップ領域は `div` の click のみで、キーボード操作やフォーカスが用意されていない（[`index.html`](index.html#L21-L25)、[`js/generator.js`](js/generator.js#L41-L55)）。
- アイコン中心のボタンや select/range に、明示的なラベルと現在状態の通知がない（[`index.html`](index.html#L27-L61)）。
- CSS でフォーカス表示の設計がなく、`.buttons div, button` がページ内の全 button へ広く作用する（[`css/style.css`](css/style.css#L62-L80)）。

**改善案**

HTML の属性を修正し、`lang`、`label`、`aria-label`、必要に応じて `aria-live` を追加する。ドロップ領域はラベル付き file input を基本にして Enter/Space でも操作可能にし、`:focus-visible` を明確に表示する。

### 4.11 低: 顔未検出時のゼロサイズを配置計算へ渡している

顔が未検出の場合、顔 Canvas は幅・高さ0になる（[`js/generator.js`](js/generator.js#L342-L355)）。一方、配置・出力計算では0による除算や剰余が発生し、`NaN` を生成しうる（[`js/generator.js`](js/generator.js#L571-L595)、[`js/generator.js`](js/generator.js#L611-L680)）。ブラウザによっては描画が単に無視されるだけでも、コードの前提として不安定である。

**改善案**

顔の幅または高さが0なら、配置更新を行わず、保存時は背景画像だけを出力する明示的な分岐を置く。Capture ボタンを「背景ロード済み」と「顔取得済み」のどちらで有効にするかも仕様として決める。

### 4.12 低: 細かな保守性の問題

- `imgCanvasCtx` の未使用代入があった。
- `drawLoop` に未使用の `useFrontCamera` 引数があった。
- `capture` の既定名が `caputure` になっていた。
- CSS に空の規則があった。
- Font Awesome を外部 URL から読み込んでいたため、通信失敗時には操作アイコンが見えなくなっていた。
